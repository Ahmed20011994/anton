1 System Overview

Anton is a product intelligence platform that answers one question every week: where is engineering attention misaligned with customer pain? It does this by pulling data from support tools (Zendesk) and engineering tools (Jira), finding the patterns customers are reporting, matching them to what engineering is working on, and surfacing the gaps — shipped without being felt, fixed but still breaking, building while customers scream about something else.

The output is a weekly digest email. Not a dashboard. Not a real-time feed. A weekly document that a PM or engineering lead can read in five minutes and act on.

2 The Architectural Rule

The intelligence layer must never know which tools the data came from. Connectors translate external tools into canonical models. The intelligence layer reads canonical models only. Adding a new tool means writing a new connector — not touching the intelligence layer.

This rule has one practical consequence: if you search the entire pipeline/intelligence/ directory, you will find zero imports from pipeline/connectors/, zero references to "zendesk" or "jira", and zero field accesses like ticket.type or issue.issuetype. The intelligence layer knows three things: CustomerSignal, WorkItem, and Account. That is its entire vocabulary.

The practical payoff: when a customer says "we use Intercom, not Zendesk", the answer is "we're writing that connector" — not "we need to rewrite the pipeline". The canonical model abstraction is what makes this a multi-tool platform rather than a Zendesk+Jira script.

3 System Layers

Layer 1

Connectors

Each integration is a standalone module. Authenticates with an external tool, pulls data incrementally, and maps it into canonical models. Handles its own pagination, rate limits, retries, and errors. Does not interpret or score data. If one connector fails, the others still run.

Layer 2

Canonical Data Layer

The internal language the platform speaks regardless of source. A Zendesk ticket and an Intercom conversation both become CustomerSignal. A Jira issue and a Linear issue both become WorkItem. Stored in PostgreSQL with pgvector for semantic search.

Layer 3

Intelligence Pipeline

Operates on canonical models only. Runs weekly. Embeds, clusters, labels, matches, scores, and detects misalignment signals. Each step is a standalone module. New detectors can be added without modifying existing ones.

Layer 4

Delivery

Renders the output of the intelligence pipeline into a weekly digest. Markdown or HTML, delivered by email. The digest template is configurable. No dashboard. No real-time feed. Thin layer — its only job is formatting and sending.

4 Layer 1 — Connectors

Connectors are the boundary between the outside world and the platform. Everything messy about external APIs — auth flows, cursor pagination, rate limit headers, retries, schema drift — is handled here and never leaks past this layer.

Connector Interface

Every connector implements the same abstract base class. The orchestration layer calls them identically regardless of source. The interface is defined in pipeline/connectors/base.py.

from abc import ABC, abstractmethod
from datetime import datetime
from typing import Iterator

class ConnectorBase(ABC):

    @abstractmethod
    def connect(self) -> None:
        """Validate credentials. Raise ConnectorAuthError on failure.
        Must be idempotent — safe to call multiple times."""

    @abstractmethod
    def health_check(self) -> dict:
        """Return {"ok": bool, "latency_ms": int, "detail": str}.
        Must never raise — surface errors in the returned dict."""

    @abstractmethod
    def sync(self, since: datetime) -> Iterator[CustomerSignal | WorkItem | Account]:
        """Yield canonical models updated since `since`.
        Handles pagination, rate limiting, and retries internally.
        Yields as items are fetched — do not buffer in memory."""

    @abstractmethod
    def backfill(self, from_date: datetime) -> Iterator[CustomerSignal | WorkItem | Account]:
        """Full historical pull. Same contract as sync()."""


# Error hierarchy
class ConnectorError(Exception): ...
class ConnectorAuthError(ConnectorError): ...
class ConnectorNetworkError(ConnectorError): ...
class ConnectorRateLimitError(ConnectorError):
    def __init__(self, retry_after_seconds: int): ...

Reliability & Error Handling

The ingestion runner wraps each connector in an isolated try/except so a Zendesk outage never blocks Jira ingestion:

def run_nightly_sync(tenant_id: str, conn) -> SyncSummary:
    results = {}
    for connector in [ZendeskConnector(), JiraConnector()]:
        try:
            connector.connect()
            since = _get_last_sync_time(conn, tenant_id, connector.source_type)
            count = 0
            for record in connector.sync(since=since):
                writer.upsert(conn, tenant_id, record)
                count += 1
            results[connector.source_type] = {"ok": True, "count": count}
        except ConnectorError as exc:
            results[connector.source_type] = {"ok": False, "error": str(exc)}
            # log and continue — do not re-raise
    return results
Idempotency guarantee: All upserts use INSERT ... ON CONFLICT (tenant_id, source_type, source_id) DO UPDATE SET .... Running the same sync twice produces identical DB state. A failed mid-run sync is safe to retry without cleanup.

The embeddings table uses an IVFFlat index for approximate nearest-neighbour search:

Similarity search used by the matching step:

SELECT entity_id, 1 - (embedding <=> %s) AS cosine_sim
FROM embeddings
WHERE tenant_id = %s AND entity_type = 'work_item'
ORDER BY embedding <=> %s
LIMIT 10;

Migration Strategy

Migrations are raw SQL files in migrations/, named numerically (001_create_extensions.sql, etc.). Python owns migrations — the Go server never runs them. pipeline/db.py:run_migrations() is called at the start of every entrypoint. It reads schema_migrations to determine which files have been applied, applies outstanding ones in order, and records them. This is intentionally simple — no Alembic, no down migrations in v1.

7 Layer 3 — Intelligence Pipeline

The pipeline is orchestrated by pipeline/intelligence/pipeline.py:run_weekly_pipeline(tenant_id, conn). It creates a digest_runs row with status='running', then chains seven steps. Any unhandled exception sets status='failed' and records error_message. The pipeline produces a digest with partial data rather than failing entirely.

1

Embed — intelligence/embed.py

Generates vector embeddings for all customer signals and work items whose content_hash has changed since last embedding. Batches 100 records per API call to text-embedding-3-small. Skips unchanged records entirely. Records token usage and cost estimate in cost_ledger. Runs nightly (not just weekly) so the weekly pipeline always finds fresh embeddings.

2

Cluster — intelligence/cluster.py

Loads all signal embeddings for the tenant as a numpy array. Runs HDBSCAN with min_cluster_size=5, min_samples=3, metric='cosine'. Noise points (label -1) are tracked but excluded from labeling. Inserts clusters rows and signal_cluster_memberships rows with soft membership probabilities. Returns a list of ClusterRecord objects containing the cluster ID, signal IDs, and centroid vector.

3

Label — intelligence/label.py

For each cluster, takes the top 10 signals by membership probability and sends their subjects and bodies to GPT-4o with a structured JSON output prompt requesting a 3–5 word theme label and a one-sentence summary. Updates clusters.label and clusters.summary. Records cost. LLM is used here because labeling is genuinely a language understanding task that SQL cannot do.

4

Match — intelligence/match.py

For each cluster centroid, performs a pgvector cosine similarity search against work item embeddings. Threshold: cosine_sim ≥ 0.75. Also checks explicit links: if any signal in the cluster has a source_id that appears in any work item's linked_customer_signals, that work item is matched with match_method='explicit_link' regardless of similarity score. Inserts work_item_matches rows.

5

Score — intelligence/score.py

Computes a composite relevance score per cluster: cosine_sim × signal_volume_weight × account_tier_multiplier. Signal volume weight is the cluster's signal count normalized against the max signal count this run. Account tier multiplier is derived by looking up the accounts of signals in the cluster — enterprise accounts get a higher weight. Used to rank themes and weight misalignment findings.

6

Detect — intelligence/detectors/

Six standalone detector functions, each called independently. Each returns a list of DetectorResult objects with severity, headline, body, and supporting data. Detectors are isolated — adding a new one never modifies an existing one. All findings are collected and passed to the renderer.

Misalignment Detectors

coverage_gap

Coverage Gap

A cluster is generating significant customer signal volume but has no active work items addressing it. Engineering is unaware of or ignoring the issue.

Clusters with signal_count above threshold AND zero rows in work_item_matches for this run_id.
silent_ship

Silent Ship

A work item was shipped, but the customer signal volume for the related cluster did not decrease after the ship date. The fix didn't actually fix it from the customer's perspective.

Work items where status_category='done' and closed_at is within 60 days. Compare signal count in the 2 weeks before vs. 2 weeks after closed_at. Flag if post-ship volume ≥ pre-ship volume.
fixed_not_fixed

Fixed Not Fixed

A work item is marked closed, but new signals semantically similar to that work item continue to be created after the close date. Customers are still hitting the bug.

Work items with closed_at set. Count new customer_signals with created_at > closed_at whose embedding cosine similarity to the work item embedding is ≥ 0.80. Flag if count > 3.
attention_asymmetry

Attention Asymmetry

Engineering activity is concentrated on clusters with low customer signal volume, while high-signal clusters have little or no engineering attention. Effort is inverted relative to customer pain.

Rank clusters by signal_count DESC. Rank clusters by active work item count DESC. Flag clusters where signal rank is in top quartile but work item rank is in bottom half.
forgotten_backlog

Forgotten Backlog

Work items that have explicit links to customer signals but have not been touched in months. Someone acknowledged the customer pain and created a ticket — then it was never picked up.

Work items where linked_customer_signals is non-empty AND updated_at < NOW() - INTERVAL '90 days' AND status_category != 'done'.
release_correlation

Release Correlation

A spike in customer signals appears within a short window after a set of work items are closed under the same version or release tag. A release introduced a regression.

Group work items by version where multiple items closed in same 7-day window. Measure customer_signal.created_at density in 7 days post-release vs. baseline. Flag if spike > 2σ above baseline.

DetectorResult interface

from dataclasses import dataclass

8 Layer 4 — Delivery

The delivery layer is intentionally thin. Its only jobs are: render the intelligence output into a readable document, and send it. No business logic lives here.

Digest Renderer — pipeline/delivery/renderer.py

render_digest(run_id, tenant_id, conn, findings: list[DetectorResult]) -> str reads the digest_run, clusters, matches, and findings, and produces Markdown. The rendered string is stored in digest_runs.digest_markdown before being sent, so every digest is permanently archived.

Go Server Role

In v1, the Go server serves only GET /healthz. There is no API between Go and Python. Python writes directly to PostgreSQL. The two processes are fully decoupled — the Go server can be down without affecting pipeline runs. The Go server's presence is forward-looking: it is the foundation for future webhooks, a configuration API, and per-user digest endpoints when those are needed.

No API between Go and PythonNo coupling means either can be changed independently. The Go server is a foundation, not a dependency.

The Correctness Test

After v1, imagine adding Intercom as a third connector and onboarding a second customer who uses it. If that requires only (1) writing a new connector module and (2) creating a new tenant row — the architecture is correct. If it requires touching the intelligence layer, the scoring logic, or the digest renderer — the architecture has failed.
